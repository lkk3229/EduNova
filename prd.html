1. Product Vision

1.1 Vision Statement

"Learn Without Limits" — EduNova is an inclusive, all-in-one online education platform that makes quality education accessible to every learner, from playway toddlers to PhD researchers, across all major education boards and professional disciplines, in their local currency and language of instruction.

1.2 Mission

To bridge educational gaps by providing a single platform where students of any age, any board, and any career stage can find expert-led courses — self-paced or live — earn verifiable certificates, and where teachers can build a sustainable income by sharing their knowledge with a global audience.

1.3 Problem Statement

  • Fragmentation: Students must navigate multiple platforms for different education levels (K-12 vs college vs professional) and different boards (CBSE, ICSE, IB, State boards).
  • Pricing Barriers: Most global ed-tech platforms price in USD, creating affordability barriers for students in developing countries.
  • Completion Accountability: Self-paced courses have low completion rates; live courses lack structured completion workflows.
  • Certificate Credibility: Many platforms offer generic certificates without unique verification IDs or URLs.
  • Teacher Monetization: Teachers often lack transparent revenue-sharing models.

1.4 Solution

EduNova solves these problems through:

  • A unified 120+ course catalog spanning playway to PhD and 12+ professional categories
  • 18-currency localization with timezone-based auto-detection (default INR)
  • Dual completion model — self-completion for recorded courses, teacher/admin-gated completion for live courses
  • Verifiable certificates with unique ENOVA-XXX-XXX IDs and verification URLs
  • Transparent 60/40 revenue split favoring teachers

6.2 Feature Details by Module

6.2.1 Course Catalog Engine

  • 120 courses across 9 education levels and 12+ categories
  • 5-axis filtering: education level, category, difficulty, price range, rating
  • All boards covered: CBSE, ICSE, State Boards, IB, Cambridge, IGCSE
  • Professional categories: Web Dev, AI/ML, Cloud, Cybersecurity, Data Science, DevOps, Mobile Dev, Blockchain, UI/UX
  • Each course displays: title, instructor, price (localized), original price, rating, student count, duration, lectures, badge

6.2.2 Certificate Generation Engine

  • HTML5 Canvas rendering to PNG
  • Fields: Certificate Name, Provider (EduNova), Student Name, Course, Teacher, Completion ID (ENOVA-XXX-XXX), Verification URL, Issue Date
  • Download as PNG file
  • Preview modal with all details
  • Admin can view all certificates and revoke

6.2.3 Revenue Engine

  • 60% teacher share / 40% platform share
  • Revenue tracked per enrollment amount
  • Teacher dashboard shows earnings breakdown
  • Admin dashboard shows all teacher revenues
  • CSV export with split amounts

6.2.4 English Speaking Trainer Engine

  • Dedicated English speaking module for student role with login-gated entry and redirect handling
  • SpeechRecognition for live listening transcript and SpeechSynthesis for spoken tutor responses
  • Question-first tutor behavior: direct answer first, then correction/tip for language improvement
  • Pronunciation scoring via phoneme-level similarity and Levenshtein-based comparison logic
  • Per-student scorecard with fluency average, pronunciation average, practice streak, and suggestions
  • Configurable LLM provider support (OpenAI/Anthropic/custom endpoint) with local fallback trainer
  • Conversation-focused response style with human-like follow-up prompts to sustain dialogue

6.2.5 Library Module Engine

  • Dedicated Library page with board-wise (NCERT, ICSE, State Boards) and class-wise (1-12) segmentation
  • Search and filter layer over catalog by title, subject, stream, class, and board
  • Book preview panel showing chapter-level and concept-level summaries
  • Library AI assistant with catalog-first responses and built-in educational fallback when no exact catalog match is found
  • Navigation integration in public navbar and logged-in user dropdown for faster access
  • Subscription model intentionally deferred for later business decision

6.2.6 Authentication Recovery Engine

  • Forgot password action from login page with inline modal/user-input flow
  • Password reset persists to registered user records in local storage
  • Demo credentials intentionally excluded from reset path to preserve demo integrity

6.2.7 Backend Runtime Reliability

  • Express error middleware is mounted through named export binding to avoid invalid middleware registration at startup.
  • Environment validation blocks server boot on invalid configuration, enabling deterministic failure diagnostics before API traffic.
  • This reliability guard directly supports end-to-end verification for checkout, verify, and media upload APIs.

6.2.8 Production Deployment Baseline

  • Dedicated production env template file (backend/.env.production.example) defines required values and safe placeholders.
  • Deployment operations guide now includes a Phase 5 production checklist across infrastructure, secrets, security, and runtime verification.
  • Checklist requires go-live validation for payment keys, email credentials, health checks, and rollback readiness.

6.2.9 Containerization and CI Baseline

  • Backend now ships with backend/Dockerfile, backend/.dockerignore, and backend/docker-compose.production.yml.
  • GitHub Actions workflow (.github/workflows/backend-ci.yml) runs backend smoke checks for pushes and pull requests.
  • backend/scripts/ciSmoke.js validates JavaScript syntax, required deployment artifacts, and env template key completeness.

6.2.10 Audit Logging and Governance

  • Audit middleware records mutation requests on privileged routes (/api/admin, /api/payments, /api/uploads, /api/monitoring).
  • Sensitive request fields are redacted before audit persistence to reduce secret exposure risk.
  • Feature is configurable through AUDIT_LOGGING_ENABLED to support staged rollout and operational tuning.

6.2.11 PostgreSQL Backend via Supabase (Phase 8)

  • Supabase integration is available through backend/config/supabase.js and repository adapters, but production route execution remains Mongo-backed until migration is complete.
  • Repository pattern currently includes backend/repositories/UserRepository.js and can be extended per route domain.
  • 10 PostgreSQL tables with proper indexes: users, courses, enrollments, certificates, books, borrowing, meetings, meeting_attendees, transactions, audit_logs.
  • Row-Level Security (RLS) policies enable per-user data access control at the database layer.
  • Environment setup now uses DB_MODE contract (mongo, supabase, hybrid) with explicit required variables per mode.
  • MongoDB is the current production dependency while core routes still use Mongoose models.
  • Supabase Realtime subscriptions enable live updates without polling (future phase).

6.2.12 Deployment Assurance Gates (Phase 8.1)

  • Health endpoint now returns live DB connectivity payload and readiness boolean.
  • Readiness endpoint (/api/ready) returns 503 until required DB dependencies are connected.
  • CI workflow provisions Mongo service, starts backend with CI env, and runs critical path integration checks.
  • Production rehearsal script executes compose startup, health checks, backup/restore, and backend recreate rollback simulation.
  • Env templates now ship without live Supabase secrets; key rotation is mandatory after exposure events.

  • localStorage limit: ~5-10 MB per origin depending on browser
  • Gateway optional mode: If payment keys are missing, checkout falls back to manual/dev completion mode
  • No distributed media store: Uploads are stored on local filesystem by default
  • No distributed cache layer yet: Response cache is process-local and non-shared across instances
  • Single admin: One super admin account (multi-admin not yet supported)
  • English only: UI is English; multi-language support is Phase 6

14.3 Assumptions

  • Users have a modern browser (Chrome 80+, Firefox 78+, Safari 14+, Edge 80+)
  • JavaScript is enabled in the browser
  • Users have internet access (for CDN resources on first load)
  • The platform is used as a prototype/demo until Phase 5 server-side deployment
