1. Platform Overview

EduNova is a full-featured online education platform built entirely with HTML5, CSS3, and vanilla JavaScript (no frameworks). It supports students from playway to PhD level and covers all major education boards and professional fields.

1.1 Core Value Proposition

  • 120+ courses spanning 8 education levels (Playway to PhD) and 12+ categories
  • Support for all major education boards — CBSE, ICSE, State Boards, IB, Cambridge, IGCSE
  • Multiple teaching modes — Self-paced (recorded), 1-on-1 live, group live, hybrid
  • 18-currency localization with auto-detection by timezone
  • Certificate generation on course completion with unique IDs and verification URLs
  • AI chatbot with confidentiality-first design (40+ KB knowledge base)
  • Revenue sharing — 60% teacher / 40% platform

  • Session object: { email, name, role, loggedIn: true, loginTime }

  • Students: Register via register-student.html — immediate access after login
  • Teachers: Register via register-teacher.html — status set to "pending", must be approved by admin
  • Passwords stored in localStorage user objects and verified on login
  • Password strength meter with 5 levels (Very Weak → Very Strong)
  • Minimum 8 characters required

  • admin.html: Redirects to login if session role ≠ 'admin'
  • student-dashboard.html: Redirects to login if session role ≠ 'student'
  • teacher-dashboard.html: Redirects to login if session role ≠ 'teacher'

  • Rejected teachers see rejection message

  • Clears edunova_session from localStorage
  • Redirects to login.html

3.6 Password Reset

  • Forgot password flow available on login page with email verification and new password entry
  • Demo accounts are excluded from reset and must use fixed demo credentials
  • Updated password is persisted in edunova_users localStorage records

6. Enrollment & Payment

6.1 Enrollment Flow

  1. Student clicks "Enroll Now" on course card (courses.html)
  2. Payment modal opens showing price in user's detected currency
  3. Frontend calls POST /api/payments/checkout/:courseId with JWT authorization
  4. If Razorpay keys exist, order is created and pending transaction is stored
  5. If gateway keys are not configured, manual/dev fallback marks transaction as completed
  6. Enrollment is persisted in MongoDB and attached to student and course records
  7. Student redirected to dashboard

6.2 Payment API Contracts

Backend startup now requires valid middleware registration ordering and references before accepting traffic.

  • Central error middleware is mounted using named export wiring from backend/middleware/errorHandler.js to avoid startup-time middleware type failures.
  • Environment validation executes before service boot, so misconfiguration fails fast with explicit diagnostics.

12.5 Production Deployment Checklist (Phase 5)

A production checklist and environment baseline now exist to standardize go-live preparation.

  • backend/.env.production.example defines required production env keys with safe placeholders.
  • backend/DEPLOYMENT_GUIDE.md includes infrastructure, secret management, security, and runtime verification checklist items.
  • Checklist explicitly validates payment gateway keys, email credentials, monitoring readiness, and rollback planning.

12.6 CI/CD and Container Operations (Phase 6)

The backend now includes built-in CI and containerization assets for consistent deployments.

  • backend/scripts/ciSmoke.js runs syntax and env-template completeness checks via npm run test:ci.
  • .github/workflows/backend-ci.yml executes backend smoke checks in GitHub Actions using Node 20.
  • backend/Dockerfile and backend/docker-compose.production.yml provide reproducible runtime packaging and startup orchestration.

12.7 Database Migration to Supabase (Phase 8)

The backend now supports PostgreSQL via Supabase as an integration track, while MongoDB remains the production runtime dependency until route migration is complete.

  • backend/config/supabase.js initializes Supabase PostgreSQL client with SERVICE_KEY authentication and auto-testing on startup.
  • backend/repositories/UserRepository.js implements repository pattern for all user database operations (find, create, update, delete, list, check exists).
  • backend/scripts/supabaseSchema.sql creates 10 PostgreSQL tables: users, courses, enrollments, certificates, books, borrowing, meetings, meeting_attendees, transactions, audit_logs. Includes indexes, RLS policies, and auto-update triggers.
  • SUPABASE_INTEGRATION.md provides step-by-step setup, schema creation, environment configuration, and migration checklist.
  • Environment variables: DB_MODE, MONGODB_URI, SUPABASE_URL, SUPABASE_ANON_KEY, SUPABASE_SERVICE_KEY now follow an explicit DB contract (mongo|supabase|hybrid).
  • Readiness endpoint (/api/ready) now verifies actual DB connectivity state and returns 503 until required dependencies are connected.
  • Row-Level Security (RLS) policies auto-created for data isolation: users see themselves (admins see all), courses public-read with instructor-edit, enrollments student-scoped, audit logs admin-only.

12.8 CI Contract and Production Rehearsal (Phase 8.1)

Release quality gates now include DB-mode validation, critical path API checks, and scripted production-like rehearsal.

  • backend/scripts/ciSmoke.js validates DB_MODE and env contract alignment for both development and production templates.
  • .github/workflows/backend-ci.yml provisions Mongo service, boots backend, and executes npm run test:integration:critical.
  • backend/scripts/rehearseProduction.js runs compose startup, health/readiness checks, backup/restore rehearsal, and backend recreate rollback simulation.

13. Security Measures

13.1 Authentication Security

  • Password verification for all registered users (email + password match)
  • Session-based access control on all dashboards
  • Automatic redirect to login if session invalid or wrong role
  • Super admin account protected from deletion

13.2 Input Security

  • HTML escaping via escapeHtml() function (creates text node, reads innerHTML)
  • Chatbot input sanitized before processing
  • Form validation on all registration and login forms

13.3 Data Confidentiality

  • Chatbot BLOCKED_KEYWORDS prevents exposure of: passwords, account numbers, SSN, credit cards, bank details, salary, admin credentials, API keys, private keys
  • No sensitive data displayed in chatbot responses
  • Session cleared on logout
  • Audit logs redact sensitive request fields (password, token, secret, signature, key, authorization, cookie) before persistence

13.4 Known Limitations

  • Data stored in localStorage (client-side only, no server encryption)
  • Passwords stored in plaintext in localStorage (acceptable for demo; production would use bcrypt + server-side storage)
  • No CSRF protection (no server-side state)
  • Demo credentials are hardcoded in auth.js
  • Current response caching is in-memory per process (no shared Redis layer across multiple instances yet)

16. Progressive Web App (PWA)

16.1 Overview

EduNova is a fully installable Progressive Web App on both Android and iOS. No app store submission required. Users install directly from their browser.

16.2 PWA Components

16.3 Installation Flow

PlatformTriggerFlow
Android (Chrome)beforeinstallprompt eventFloating banner appears → user clicks Install → native Chrome prompt → accepted → app icon added to home screen
iOS (Safari)Manual (Apple restriction)User clicks iPhone App button → modal shows step-by-step: Share → Add to Home Screen → Add → app icon added
Desktop (Chrome/Edge)beforeinstallprompt eventSame as Android — browser install prompt in address bar or via Install button

16.4 Offline Capability

  • Full app shell cached on first visit (all HTML, CSS, JS pages).
  • Dynamic assets cached as visited and served from cache when offline.
  • API calls fail gracefully with JSON error message when offline.
  • HTML navigation falls back to index.html when network unavailable.

16.5 Homepage Integration

  • "Get the App" section added before Contact on index.html — contains Android install button, iPhone button, phone mockup, floating feature badges.
  • Floating install banner shown automatically when Android install is available (pwa-banner-visible class added by pwa.js).
  • Footer has Android App and iPhone App buttons in a dedicated Get the App column.
  • All install buttons trigger window.installPWA() (Android) or window.showIOSInstall() (iOS modal).
